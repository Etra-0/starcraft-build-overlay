## Code of conduct

Be respectful, assume good intent, and keep discussions focused on the project. Bug reports and feature requests are welcome — please include reproduction steps, OS/version info, and screenshots.

## Development environment

### Prerequisites

- **Node 24 LTS** (use [nvm](https://github.com/nvm-sh/nvm) / [nvm-windows](https://github.com/coreybutler/nvm-windows); the repo has a [`.nvmrc`](.nvmrc), so `nvm use` picks the right version automatically).
- **npm 11+** (ships with Node 24).
- **Rust 1.85+** via [`rustup`](https://rustup.rs/). The [`rust-toolchain.toml`](rust-toolchain.toml) at the repo root pins `stable` and ensures `clippy` + `rustfmt` are installed on first `cargo` invocation.
- **Git**.
- **OS-specific Tauri build deps:**
  - **Windows:** Microsoft Visual C++ Build Tools (workload "C++ build tools"). `rustup-init` will offer to install them; pick the MSVC toolchain. WebView2 is pre-installed on Win10 21H2+ and Win11.
  - **macOS:** `xcode-select --install`.
  - **Linux:** `libwebkit2gtk-4.1-dev`, `libgtk-3-dev`, `libsoup-3.0-dev`, `libjavascriptcoregtk-4.1-dev`, `libayatana-appindicator3-dev`, `librsvg2-dev` (Debian/Ubuntu names; see [Tauri's prereq page](https://v2.tauri.app/start/prerequisites/#linux) for dnf/pacman).

Verify:

```bash
node --version   # v24.x
npm --version    # 11.x or newer
rustc --version  # 1.85 or newer
cargo --version
```

VS Code users: `.vscode/extensions.json` recommends `rust-lang.rust-analyzer` and `tauri-apps.tauri-vscode` — install both for inline diagnostics on the Rust side.

### One-time setup

```bash
git clone https://github.com/Etra-0/starcraft-build-overlay.git
cd starcraft-build-overlay
npm install
```

`npm install` pulls the small JS-side toolchain:

- **@tauri-apps/cli** (drives `tauri dev` / `tauri build`)
- **@tauri-apps/api** + plugin packages (typed JS bindings — used at runtime by `src/renderer/api.ts`)
- **esbuild** (bundles the renderer)
- **typescript**, **@typescript-eslint**, **eslint** (flat config), **prettier**

`cargo` will fetch the Rust crates lazily on the first `cargo build` / `npm run tauri:dev`. The toolchain pin in `rust-toolchain.toml` triggers a quick `rustup` component install on the first run.

## Daily workflow

```bash
npm run dev
```

`npm run dev` runs two things in parallel via `scripts/dev.mjs`:

1. `esbuild --watch` rebuilds the renderer bundle into `dist-frontend/` whenever a `src/renderer/` file changes.
2. `npx tauri dev` builds and launches the Rust backend with `BW_DEVTOOLS=1` so DevTools auto-opens.

Renderer changes hot-reload on save. **Rust changes** trigger a recompile + relaunch automatically (Tauri watches `src-tauri/`).

### Faster iteration on a single layer

```bash
npm run build:renderer       # one-shot renderer rebuild
npm run dev:renderer         # renderer-only watch (no Tauri)
cargo check --manifest-path src-tauri/Cargo.toml   # fast Rust syntax/borrow check
cargo test  --manifest-path src-tauri/Cargo.toml   # Rust unit tests
```

### Cleaning everything

```bash
npm run clean   # deletes dist-frontend/ and src-tauri/target/
```

`dist-frontend/` and `src-tauri/target/` are git-ignored and regenerated by every build. Deleting them is always safe — `npm run build:renderer` and `npm run tauri:build` rebuild from scratch.

## Debugging

### In-app DevTools

- **`F12`** or **`Ctrl+Shift+I`** toggles DevTools at any time, in any build (dev or packaged).
- DevTools opens automatically when `BW_DEVTOOLS=1` is set, which `scripts/dev.mjs` does for you in dev mode. To force-open in a packaged build:
  ```powershell
  # Windows
  $env:BW_DEVTOOLS = "1"
  & "$env:ProgramFiles\BW Build Overlay\BW Build Overlay.exe"
  ```
  ```bash
  # macOS / Linux
  BW_DEVTOOLS=1 "BW Build Overlay"
  ```
- The Console tab shows logs. The Network tab shows Liquipedia requests. The Sources tab lets you set breakpoints in the bundled renderer (a source map is emitted next to `dist-frontend/renderer.js`).

If the renderer fails to boot at all, a red banner appears at the top of the window with the error message. The DevTools console will have the full stack.

### Rust-side logs

Backend `log::*` / `eprintln!` / `println!` calls write to **stdout/stderr of the launched process**:

- In dev mode that's the terminal you ran `npm run dev` in.
- In a packaged build: launch from a terminal (`./BW\ Build\ Overlay` on Linux, `open -a "BW Build Overlay" --stdout out.log --stderr err.log` on macOS, or `& 'C:\...\BW Build Overlay.exe' 2>&1` in PowerShell on Windows) to see them.

### Reset the user data

The app stores builds and settings under `<app_data_dir>/com.local.bwbuildoverlay/`:

```powershell
# Windows
Remove-Item "$env:APPDATA\com.local.bwbuildoverlay" -Recurse -Force
```

```bash
# macOS
rm -rf "$HOME/Library/Application Support/com.local.bwbuildoverlay"

# Linux
rm -rf "${XDG_DATA_HOME:-$HOME/.local/share}/com.local.bwbuildoverlay"
```

**Manage > Backup builds.json** in the UI saves a timestamped `builds-backup-*.json` first if you want to keep your library.

## Testing

```bash
npm test
```

`npm test` is `cargo test --manifest-path src-tauri/Cargo.toml` — runs all 19 Rust unit tests against `src-tauri/src/`:

- `storage::tests::*` — JSON I/O, schema migration, settings round-trip.
- `liquipedia::parser::tests::*` — wikitext parser coverage of the `{{build}}` template + infobox.
- `liquipedia::api::tests::*` — title/URL parsing.
- `liquipedia::import::tests::*` — variant-linking + curated-title detection.
- `utils::tests::*` — `derive_matchup`, `slugify`, `split_step` parity with `src/shared/utils.ts`.

To add a Rust test, drop a `#[test]` (or `#[tokio::test]` for async) into the relevant module's `mod tests` block. Use `tempfile::tempdir()` for any fs-touching test so the suite stays parallelizable.

All tests live in Rust — see [Testing](#testing). The renderer is small and DOM-driven; the meaningful storage / parser / importer logic is in `src-tauri/src/` where it can be exercised directly with `cargo test`.

### Quality checks

```bash
npm run typecheck       # tsc on the renderer
npm run lint            # eslint over **/*.ts
npm run format:check    # prettier --check .
cargo fmt --all --check --manifest-path src-tauri/Cargo.toml
cargo clippy --all-targets --manifest-path src-tauri/Cargo.toml -- -D warnings
cargo test  --manifest-path src-tauri/Cargo.toml
```

CI is split into two jobs:

- **quality** (`ubuntu-latest`): TS quality (format / lint / typecheck) + the four Rust commands above. Fastest signal for the typical "did you break something" path.
- **package** matrix (`windows-latest` + `macos-latest` + `ubuntu-latest`, `needs: quality`): builds the renderer and runs `cargo build --tests` so platform-specific Rust regressions surface on PRs without paying for the full bundle build.

Both are triggered on push-to-`main` and on `pull_request`. Concurrency is keyed on the ref so a new push to a branch cancels the previous in-progress run instead of queuing.

## Code style

### TypeScript (renderer)

- Strict mode is on across `src/`. **No `// @ts-nocheck` directives** — new files should not introduce them.
- The IPC contract lives in [`src/shared/types.ts`](src/shared/types.ts) (`OverlayAPI` interface). Adding a new IPC channel means: declare it in `OverlayAPI`, add a `#[tauri::command]` in [`src-tauri/src/commands.rs`](src-tauri/src/commands.rs), wire it into the `invoke_handler!` macro in [`src-tauri/src/lib.rs`](src-tauri/src/lib.rs), grant any required permission in [`src-tauri/capabilities/default.json`](src-tauri/capabilities/default.json), and consume it via `api.X(...)` from [`src/renderer/api.ts`](src/renderer/api.ts). All four sides are typecheck-linked; if you forget the renderer side, `npm run typecheck` fails.
- Renderer DOM lookups go through [`src/renderer/dom.ts`](src/renderer/dom.ts)'s `byId<T>` helper, which throws on a missing id rather than silently returning `null`.
- Adding a new global hotkey: extend the `HotkeyAction` union in `src/shared/types.ts`, register it in [`src-tauri/src/window.rs`](src-tauri/src/window.rs) `register_shortcuts`, emit `app.emit("hotkey", action)` from the callback, and add the matching `case "..."` to the `switch` in [`src/renderer/hotkeys.ts`](src/renderer/hotkeys.ts). The TS `never`-check default and Rust's `match` exhaustiveness both catch missed branches at compile time.
- Pure helpers consumed by both the renderer (TS) and the Rust backend should be implemented in **both** layers and kept behavior-identical with a Rust unit test (see `utils.rs`); the TS copy in `src/shared/utils.ts` exists because the renderer can't call Rust synchronously.
- Prettier handles formatting (`npm run format`).
- ESLint runs the typescript-eslint recommended ruleset plus `switch-exhaustiveness-check`, `eqeqeq` (with `null: ignore`), and `prefer-const`.

### Rust (backend)

- `cargo fmt` is canonical — run `cargo fmt --all` before pushing.
- `cargo clippy --all-targets -- -D warnings` is enforced by CI; warnings are errors.
- `serde` structs in `src-tauri/src/types.rs` use `#[serde(rename_all = "camelCase")]` to match the JS-side JSON layout. **Don't change field names without updating the matching TS type and a `cargo test` round-trip case.**
- The `tauri::Manager` / `AppHandle` / `WebviewWindow` types are managed by Tauri — don't move them between threads or `clone()` them aggressively; pass `app.clone()` only when you need to spawn an async task.
- New error types should derive `thiserror::Error`. Convert to `String` at the IPC boundary (Tauri commands need `Result<T, String>`).

### Comments and headers

- Every source file in `src/` and `src-tauri/src/` has a 2–3 line purpose header at the top. Preserve this when refactoring; it's how new contributors orient.
- **Don't** add comments that just narrate what the code does (`// import the module`, `// loop through items`). Only document non-obvious intent or constraints.
- Imports stay at the top of the file (`no-inline-imports` rule).

## Commit messages

Use a loose [Conventional Commits](https://www.conventionalcommits.org/) style:

- `feat: add XYZ to the Manager` — user-visible new feature.
- `fix: stop overlay from freezing on Liquipedia 429` — user-visible bug fix.
- `chore: bump @tauri-apps/cli to 2.x` — no user-visible behavior change.
- `docs: clarify release flow in README` — docs only.
- `refactor(parser): extract find_templates helper` — code shape change, no behavior change.
- `test: cover bullet-only fallback parser` — adds or fixes tests.
- `chore(release): v2.0.1` — produced automatically by `scripts/release.mjs`.

The release script auto-generates the changelog and tag from the `## [Unreleased]` block; commit messages don't have to be machine-parseable.

**Single-author project rules** (also encoded in the workspace's contributor-policy rule): every commit is authored by `Etra-0`. Don't add `Co-authored-by:` trailers, "Generated with..." footers, or any AI/agent attribution lines. The commit subject + optional body are the entire message.

## Pull requests

Before opening a PR:

1. **Update [CHANGELOG.md](CHANGELOG.md)**: add an entry under `## [Unreleased]` describing user-visible changes (Added / Changed / Fixed / Removed). Pure refactors / docs / test-only / CI / tooling PRs can use an `### Internal / Tooling` bullet instead and do not warrant a release.
2. **Update [README.md](README.md)** when behavior, setup, or release flow changes.
3. **Run quality checks locally**:
   ```bash
   npm run format:check
   npm run lint
   npm run typecheck
   cargo fmt --all --check --manifest-path src-tauri/Cargo.toml
   cargo clippy --all-targets --manifest-path src-tauri/Cargo.toml -- -D warnings
   npm test
   ```
4. **Smoke-test the renderer**: run `npm run dev`, click around the Manager, hit a couple of hotkeys. The historical regression we want to never repeat is **"the UI loads but every click is a no-op"** — that was caused by the renderer being shipped as CommonJS instead of ESM, and is exactly the kind of thing a renderer-build-pipeline change can re-introduce. If you touched anything under `scripts/`, `src/renderer/api.ts`, or `index.html`, smoke-test before merging.
5. **Fill in [`.github/PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md)** — it mirrors the checklist above and prompts for a CHANGELOG classification and a rollback note.

PRs should be small and focused. Squash-merge is fine.

**Judging PR readiness:** prefer `gh pr checks --json name,bucket,state,workflow,link` over the Actions tab. It reports every PR-attached check (including non-GHA ones); `gh run list` only sees GitHub Actions. Re-check after every push — the attached check set can change between pushes.

## Releasing

Maintainers cut releases as follows. The whole flow is automated end-to-end via GitHub Actions.

### 1. Make sure `main` is green

```bash
git checkout main
git pull
npm ci
npm run lint
npm run typecheck
cargo clippy --all-targets --manifest-path src-tauri/Cargo.toml -- -D warnings
npm test
npm run tauri:build   # optional — sanity-check the host-OS bundle locally
```

### 2. Edit the changelog

Open [CHANGELOG.md](CHANGELOG.md), make sure `## [Unreleased]` reflects every user-visible change since the last release. Group bullets under **Added / Changed / Fixed / Removed**.

### 3. Bump the version + tag

Pick the right semver bump:

- `release:patch` for bug fixes only (`2.0.0` → `2.0.1`).
- `release:minor` for new features that don't break existing data / IPC (`2.0.0` → `2.1.0`).
- `release:major` for breaking changes — e.g. a new `SCHEMA_VERSION` that drops fields (`2.0.0` → `3.0.0`).

```bash
npm run release:minor   # bumps versions, rotates CHANGELOG.md, refreshes Cargo.lock, commits, tags
```

The script:

1. Verifies you're on `main` with a clean tree (use `--allow-dirty` to override).
2. Updates `version` in `package.json`, `src-tauri/Cargo.toml`, and `src-tauri/tauri.conf.json`.
3. Refreshes `src-tauri/Cargo.lock` via `cargo update -p bw-build-overlay`.
4. Renames `## [Unreleased]` to `## [X.Y.Z] - YYYY-MM-DD` and inserts a fresh empty `## [Unreleased]` block.
5. Commits as `chore(release): vX.Y.Z`.
6. Tags `vX.Y.Z`.

Use `--dry-run` first if you want to preview without touching anything:

```bash
npm run release:minor -- --dry-run
```

### 4. Push

```bash
git push origin main --follow-tags
```

The push of the `vX.Y.Z` tag triggers [.github/workflows/release.yml](.github/workflows/release.yml), which runs a Windows + macOS + Ubuntu matrix using [`tauri-apps/tauri-action@v0`](https://github.com/tauri-apps/tauri-action). Each runner produces its native bundles and uploads them to a draft GitHub Release named `BW Build Overlay vX.Y.Z`.

### 5. Verify and publish

- Check the [Actions tab](https://github.com/Etra-0/starcraft-build-overlay/actions) for green Release runs on all three runners.
- Open the new draft Release — confirm bundles are attached for all three OSes:
  - Windows: `*-setup.exe` + `*.msi`.
  - macOS: `*_universal.dmg` + `*.app.tar.gz`.
  - Linux: `*.AppImage` + `*.deb`.
- Optionally edit the Release body to copy in the relevant `CHANGELOG.md` block.
- Click "Publish release".

### Hotfix flow

For a same-day patch:

```bash
# fix the bug, commit, then:
npm run release:patch
git push origin main --follow-tags
```

That's it — no separate hotfix branch is needed for solo / small-team development.

## Repo layout cheat sheet

See the [Project layout](README.md#project-layout) section in the README for the full tree. The two folders you'll touch most often:

- **`src/renderer/`** — TypeScript UI. Bundled by esbuild into `dist-frontend/renderer.js`.
- **`src-tauri/src/`** — Rust backend. `cargo`-built and bundled into the Tauri app.

The IPC seam between them is `src/renderer/api.ts` (TS, calls `invoke()` / `listen()`) ↔ `src-tauri/src/commands.rs` (Rust, `#[tauri::command]`). Anything that needs to cross that boundary must have a matching type in `src/shared/types.ts` (TS) and `src-tauri/src/types.rs` (Rust) so JSON round-tripping stays loss-free.

Build outputs (`dist-frontend/`, `src-tauri/target/`) are git-ignored and disposable.
